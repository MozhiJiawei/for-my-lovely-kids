# 小红花软件 Use Case 全集

## 文档定位

本文从用户视角描述小红花软件首版本的完整使用场景。它承接并扩展 `docs/plans/2026-05-11-001-wish-management-use-cases.md`、任务管理和历史统计用例，重点回答家长和孩子“在什么情况下要做什么、系统应该给出什么结果”。

本文不描述技术实现细节。涉及家长控制、数据持久化、备份和部署的实现边界，请参考架构文档和部署文档。

## 用户与目标

### 孩子

孩子使用小程序看到自己的小花园、当前任务和心愿。孩子的核心目标是：

- 知道今天可以做哪些事情来获得小红花。
- 完成任务后立即看到小红花变化和花园成长。
- 用已获得的小红花兑换心愿。
- 回顾自己最近做过什么、换过什么。

### 家长

家长负责维护任务、心愿和历史记录，确保奖励规则公平、稳定、可解释。家长的核心目标是：

- 把家庭约定变成孩子看得懂的任务和心愿。
- 防止孩子误改规则、误删记录或绕过确认。
- 发现记录错误时能在当天修正。
- 忘记当天记录习惯时，能在合理时间范围内补录，不破坏每日一次规则。
- 确认孩子的成长数据不会因为换手机、重启服务或部署而丢失。

## 全局使用原则

- 小红花有两个数字：可用小红花和累计小红花。完成任务同时增加两者；兑换心愿只扣减可用小红花，不减少累计小红花。
- 任务分为“习惯”和“目标”。习惯每天可以完成一次，必要时家长可以补录最近 1 个月内漏记的习惯打卡；目标完成一次后退出当前任务。
- 心愿分为“一次性”和“可重复”。一次性心愿实现后退出当前心愿；可重复心愿可以多次兑换。
- 任务、心愿、历史记录和小红花等影响孩子状态的操作必须受家长控制保护。
- 历史记录用于解释小红花从哪里来、花到哪里去。已经发生的记录应保留当时的名称和小红花数量快照。
- 首版本以家庭单账号原型为边界，不处理多家庭、复杂成员权限或跨家庭分享。

## UC-Garden-01：进入花园首页

### 场景

孩子打开小程序，希望快速看到今天的小花园状态。

### 用户路径

- 孩子进入“花园”页签。
- 系统展示当前可用小红花、累计小红花、花园成长状态和首页心愿泡泡。
- 孩子可以看到花园是否因为累计小红花而进入新的成长阶段。
- 孩子可以点击心愿泡泡进入心愿详情。

### 预期结果

- 首页是孩子能理解的“当前状态”入口，而不是管理后台。
- 首页心愿泡泡只负责展示和跳转，不直接完成兑换，避免误触。
- 花园成长基于累计小红花，不会因为兑换心愿而倒退。

## UC-Garden-02：看到兑换后的纪念

### 场景

孩子实现了一个心愿，希望在花园里看到这个成果被留下。

### 用户路径

- 孩子或家长完成心愿兑换。
- 系统在历史中记录该次兑换。
- 花园保留与该兑换有关的纪念装饰。

### 预期结果

- 纪念装饰代表“已经实现过的心愿”，而不是当前可兑换心愿。
- 删除或修正当天错误兑换记录时，相关纪念也应随之修正。

## UC-Task-01：家长新增习惯

### 场景

家长希望把每天重复的家庭约定放进小程序，例如刷牙、喝水、阅读。

### 用户路径

- 家长进入“任务”页签。
- 家长选择新增任务。
- 系统要求通过家长控制。
- 家长填写任务名称、奖励小红花数量，并选择“习惯”。
- 家长保存。

### 预期结果

- 新习惯出现在当前任务列表。
- 孩子每天最多能完成一次该习惯。
- 当天已经完成的习惯应显示为今日已完成，第二天重新可完成。

## UC-Task-02：家长新增目标

### 场景

家长希望鼓励孩子完成一次性目标，例如学会系鞋带、第一次独立整理玩具。

### 用户路径

- 家长进入“任务”页签。
- 家长选择新增任务。
- 系统要求通过家长控制。
- 家长填写任务名称、奖励小红花数量，并选择“目标”。
- 家长保存。

### 预期结果

- 新目标出现在当前任务列表。
- 孩子完成并确认后，目标退出当前任务。
- 历史记录仍保留该目标完成时的名称和奖励快照。

## UC-Task-03：家长编辑当前任务

### 场景

家长发现任务名称、奖励数量或任务类型需要调整。

### 用户路径

- 家长进入任务详情或编辑页。
- 系统要求通过家长控制。
- 家长修改名称、奖励小红花数量或任务类型。
- 家长保存。

### 预期结果

- 当前任务列表展示修改后的规则。
- 已经产生的历史记录不被改写，仍展示当时发生时的快照。
- 如果输入为空或小红花数量不是正整数，系统拒绝保存并提示。

## UC-Task-04：家长删除当前任务

### 场景

某个任务不再适合孩子当前阶段，家长希望它从当前任务中消失。

### 用户路径

- 家长进入任务详情或编辑页。
- 系统要求通过家长控制。
- 家长删除任务。

### 预期结果

- 该任务不再出现在当前任务列表。
- 已发生的完成历史仍保留。
- 删除不是清空历史，孩子过去获得的小红花仍然可解释。

## UC-Task-05：孩子完成习惯或目标

### 场景

孩子完成了一个任务，希望领取小红花。

### 用户路径

- 孩子进入“任务”页签。
- 孩子点击可完成的任务。
- 系统记录完成，并为孩子增加小红花。
- 页面刷新当前任务状态、可用小红花和累计小红花。

### 预期结果

- 习惯在当天变为已完成，不能重复领取。
- 目标完成后退出当前任务。
- 系统应避免并发或重复点击导致重复加花。

## UC-Task-06：家长确认待确认任务

### 场景

当任务采用先提交、后确认的家庭规则时，家长需要确认孩子提交的任务。

### 用户路径

- 孩子提交任务完成。
- 家长在待确认入口看到任务提交。
- 系统要求通过家长控制。
- 家长确认一个或多个提交。

### 预期结果

- 被确认的提交进入历史。
- 小红花在确认时增加。
- 重复确认同一提交应被拒绝。

## UC-Task-07：家长补录习惯打卡

### 场景

孩子已经完成某个习惯，但当天忘记在小程序里打卡。家长希望事后补上这条记录，让统计和小红花保持真实。

### 用户路径

- 家长进入“任务”页签。
- 家长在某个习惯卡片上选择“补录”。
- 系统要求通过家长控制。
- 家长进入补录页，在日历中选择一个或多个要补录的日期。
- 系统在日历上标出该习惯已经打卡的日期，并将不可补录日期置灰。
- 系统只允许选择最近 1 个月内、早于今天、且尚未打卡的日期。
- 家长保存补录。

### 预期结果

- 补录成功后，该习惯在每个所选日期产生一条已确认打卡记录。
- 可用小红花和累计小红花按所选日期数和该习惯的奖励数量增加。
- 统计页中对应日期出现该习惯的打卡标记和明细。
- 同一习惯同一天已经完成时，补录应被拒绝，不能重复加花。
- 今天的习惯完成应回到主页按正常打卡流程处理，不能通过补录完成。
- 目标不能通过习惯补录入口补录。
- 超过最近 1 个月、今天或未来日期的补录应被拒绝。

## UC-Wish-01：家长新增心愿

### 场景

家长希望把孩子想要的奖励放进心愿池，例如周末旋转木马、新绘本、冰淇淋。

### 用户路径

- 家长进入“心愿”页签。
- 家长选择新增心愿。
- 系统要求通过家长控制。
- 家长填写标题、需要的小红花数量、心愿类型。
- 家长可以补充描述、图片和链接。
- 家长可以选择是否优先展示在花园首页。
- 家长保存。

### 预期结果

- 新心愿进入当前心愿列表。
- 当前可用心愿数量受首版本容量限制约束。
- 图片和链接作为心愿详情辅助信息，不改变兑换规则。
- 输入不合法时系统拒绝保存并提示。

## UC-Wish-02：孩子浏览首页心愿泡泡

### 场景

孩子在花园首页想看看自己正在努力兑换什么。

### 用户路径

- 孩子进入“花园”页签。
- 系统展示优先心愿和适合展示的当前心愿。
- 孩子点击心愿泡泡。
- 系统进入心愿详情。

### 预期结果

- 置顶心愿优先展示。
- 已实现的一次性心愿不再出现在首页泡泡中。
- 首页不提供直接兑换按钮。

## UC-Wish-03：孩子查看心愿详情

### 场景

孩子想知道某个心愿还差多少小红花、是不是可以重复兑换。

### 用户路径

- 孩子从首页泡泡或心愿列表进入详情。
- 系统展示标题、所需小红花、当前进度、类型、描述、图片和链接。
- 如果小红花足够，孩子可以发起兑换。

### 预期结果

- 详情页表达“还差多少”和“是否可兑换”。
- 一次性与可重复心愿的规则清楚可见。
- 图片加载失败不应阻断文字信息和兑换判断。

## UC-Wish-04：孩子直接兑换心愿

### 场景

家庭允许孩子在小程序里直接兑换，家长事后执行实际奖励。

### 用户路径

- 孩子在心愿列表或详情页点击兑换。
- 系统判断可用小红花是否足够。
- 小红花足够时，系统完成兑换并扣减可用小红花。
- 小红花不足时，系统提示不能兑换。

### 预期结果

- 兑换成功后产生历史记录。
- 兑换只减少可用小红花，不减少累计小红花。
- 一次性心愿兑换成功后退出当前心愿并取消首页优先展示。
- 可重复心愿兑换成功后仍保留在当前心愿中。

## UC-Wish-05：孩子申请心愿，家长批准

### 场景

家庭希望孩子可以表达想兑换，但最终扣花和批准由家长完成。

### 用户路径

- 孩子在心愿列表或详情页发起心愿兑换申请。
- 家长看到待批准的心愿兑换。
- 系统要求通过家长控制。
- 家长批准兑换。

### 预期结果

- 申请状态先为待批准，批准后进入已批准历史。
- 批准时检查可用小红花是否仍然足够。
- 小红花不足、心愿已失效或一次性心愿已兑换时，批准失败并提示。

## UC-Wish-06：家长编辑当前心愿

### 场景

家长想调整心愿成本、标题、展示优先级或补充详情。

### 用户路径

- 家长进入心愿详情或编辑页。
- 系统要求通过家长控制。
- 家长修改标题、所需小红花、类型、置顶、描述、图片或链接。
- 家长保存。

### 预期结果

- 当前心愿展示修改后的信息。
- 已经发生的兑换历史保留兑换时的快照。
- 已实现的一次性心愿不应被重新作为当前心愿展示，除非通过当天历史修正恢复。

## UC-History-01：查看总体统计

### 场景

家长或孩子想知道最近整体表现如何。

### 用户路径

- 用户进入“统计”页签。
- 系统展示累计小红花、本月开花和今日开花。
- 系统以月份日历展示每天的记录分布。
- 用户可以切换月份。

### 预期结果

- 总数来自真实历史账本，而不是页面临时状态。
- 本月和今日统计随月份、日期和记录变化更新。
- 没有记录的日期保持清爽的空状态。

## UC-History-02：查看某天明细

### 场景

家长想解释某天为什么小红花增加或减少。

### 用户路径

- 用户在统计页点击某个日期。
- 系统展示当天任务完成记录和心愿兑换记录。
- 每条记录展示名称快照、小红花数量、状态和时间。

### 预期结果

- 加花和扣花都能追溯到具体任务或心愿。
- 心愿兑换显示为扣减可用小红花。
- 历史记录不依赖当前任务或心愿是否还存在。

## UC-History-03：按习惯查看打卡

### 场景

家长想看某个习惯最近坚持得怎么样。

### 用户路径

- 用户在统计页选择一个习惯过滤。
- 日历标记该习惯在哪些日期完成。
- 用户点击日期查看该习惯当天明细。

### 预期结果

- 过滤只影响展示，不改变历史数据。
- 习惯的每日完成规则和日历标记一致。
- 目标不应被误展示为可持续打卡习惯。

## UC-History-04：家长修正当天任务历史

### 场景

家长发现今天某条任务记录的小红花数量记错，或误点完成。

### 用户路径

- 家长进入统计页并选中今天。
- 家长选择某条今天的任务完成记录。
- 系统要求通过家长控制。
- 家长修改小红花数量或删除该记录。

### 预期结果

- 只允许修正今天已经确认的任务记录。
- 修改数量时，可用小红花和累计小红花同步调整。
- 删除记录时，对应账本变动被撤销。
- 如果删除的是一次性目标的唯一完成记录，该目标可以恢复为当前目标。

## UC-History-05：家长修正当天心愿历史

### 场景

家长发现今天某次心愿兑换记错成本，或误点兑换。

### 用户路径

- 家长进入统计页并选中今天。
- 家长选择某条今天的心愿兑换记录。
- 系统要求通过家长控制。
- 家长修改兑换小红花数量或删除该记录。

### 预期结果

- 只允许修正今天已批准的心愿兑换记录。
- 修改成本时，可用小红花同步调整，累计小红花不变。
- 删除兑换时，对应扣花记录和纪念装饰被撤销。
- 如果删除的是一次性心愿的唯一兑换记录，该心愿可以恢复为当前心愿。

## UC-Parent-Control-01：首次设置家长控制

### 场景

家长第一次进行管理操作，需要设置家长控制。

### 用户路径

- 家长触发需要保护的操作。
- 系统提示设置 6 位数字家长口令。
- 家长设置后，系统短时间内保持已解锁状态。

### 预期结果

- 没有家长控制时，不应直接允许修改任务、心愿或历史。
- 口令格式必须是 6 位数字。
- 解锁状态只在有限时间内有效。

## UC-Parent-Control-02：再次进行家长操作

### 场景

家长在同一轮管理过程中连续编辑多个任务或心愿，希望不被频繁打断。

### 用户路径

- 家长完成一次口令验证。
- 家长在解锁有效期内继续管理操作。
- 超过有效期后再次操作需要重新验证。

### 预期结果

- 解锁缓存提升家长使用效率。
- 孩子拿到手机较久后不能继续使用已过期的管理权限。

## UC-Data-01：跨设备查看同一家庭数据

### 场景

家长和孩子可能使用不同手机打开小程序，希望看到同一份家庭数据。

### 用户路径

- 设备 A 完成任务或兑换心愿。
- 设备 B 打开小程序或刷新页面。
- 系统从服务端读取最新状态。

### 预期结果

- 业务数据以服务端为准。
- 小程序本地只保存必要的家长控制状态和请求配置，不作为业务真相来源。
- 并发操作不能导致重复加花、重复兑换或余额错误。

## UC-Data-02：服务重启后继续使用

### 场景

API 服务升级、重启或重新部署后，家庭希望数据仍然存在。

### 用户路径

- 服务端完成重启或部署。
- 用户重新打开小程序。
- 系统读取原有任务、心愿、历史和小红花余额。

### 预期结果

- 重启不丢失 SQLite 中的业务数据。
- 部署前后的数据库版本兼容。
- 用户不需要重新录入家庭数据。

## UC-Data-03：从备份恢复

### 场景

出现误操作、服务器故障或迁移失败时，家长希望恢复最近的可靠数据。

### 用户路径

- 维护者选择本地或 OSS 中的备份。
- 系统先验证备份完整性。
- 系统在恢复前创建安全备份。
- 系统恢复数据库并检查服务可用。

### 预期结果

- 恢复流程不直接复制正在运行的 SQLite 文件。
- 恢复失败时应尽可能回滚到恢复前状态。
- 用户最终看到的是一份可解释、完整的家庭数据。

## 首版本范围外

- 多家庭、多孩子、多家长的细粒度权限模型。
- 微信登录、真实身份绑定和云端账号迁移。
- 复杂奖励商城、库存、发货和支付。
- 家长远程通知、消息订阅和多人审批。
- 长期历史的复杂趋势分析和导出报表。
- 孩子自定义花园装饰和自由布置。

## 验收总口径

- 孩子能打开花园、完成任务、查看心愿、兑换或申请心愿，并看到小红花变化。
- 家长能新增、编辑、删除任务，补录最近 1 个月内的习惯打卡，新增和编辑心愿，并修正当天错误历史。
- 家长控制覆盖任务、心愿、历史和小红花相关管理操作。
- 习惯、目标、一次性心愿、可重复心愿的行为符合各自规则。
- 统计页能解释小红花的增加、扣减和日期分布。
- 服务端持久化、并发保护、迁移和备份恢复能支撑家庭数据稳定使用。
